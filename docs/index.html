For coding: an IDE, not a chat window

A holder for your code.

codehalter is an ACP coding agent that connects Zed, VS Code, or JetBrains to any OpenAI-compatible model, built for and focused on local LLMs. It has no UI of its own: your editor is the front end, not a CLI. Plan → execute → verify → document, all through the ACP bridge.

Designed to be held, not trusted blindly.

Every primitive assumes you want to stay in the loop.

Devcontainer-first

Scaffolds a throwaway devcontainer, Alpine, Arch, Debian, Fedora, or Ubuntu, and runs entirely inside it, so edits and build/test commands stay off your host. It works with your real .git and identity, but is instructed never to commit or push on its own, that stays your call.

Reviewable diffs

Edits land as diffs you approve in Zed. Interactive mode gates every file write and command; Autopilot runs the batch hands-free when you trust it.

Self-verify, then replan

Each subtask carries its own verify recipe and runs it before declaring done. A failure feeds a fresh plan with the context, up to 20 attempts, bailing early when the model keeps rephrasing the same problem.

Prefix-cache discipline

The rendered prompt stays byte- and position-stable turn over turn, so your server's KV cache stays ~90%+ warm. The prefix is pre-warmed at session open (first turn skips the prompt-processing wait), and even a single-slot server stays coherent: background work rides the conversation's own prefix. The only deliberate break is a new MCP server coming online.

Code intelligence, every language

Real go-to-definition, find-references and symbol search, not just for Go. gopls is wired in out of the box; TypeScript, C/C++ (clangd), Rust, Python, … bridge their LSP to MCP via lsmcp. Bring any other MCP server via mcp.toml. (non-Go bridges experimental)

Smart background compaction

Every turn, the background condenses that turn into structured notes while you read the reply, so when the context nears the model's n_ctx, compaction is already prepared and fast. With ≥2 slots it runs in parallel; on a single slot it rides the conversation's own prefix cache.

Slash-command macros

Type /grill-me and the agent interviews you about a plan until the design is resolved; /commit drafts a message from your diff and commits (never on its own initiative). Any /<name> expands a TEMPLATE-<name>.md into a full prompt, drop your own for reusable team workflows.

Sets up new projects

Point it at a fresh repo and it onboards the whole thing: scaffolds the devcontainer, detects the stack, seeds skills, and offers one-click fix cards to install what's missing, gopls/node/clang, an LSP bridge, a broken mcp.toml, doing it for you when you accept.

Learns from its own sessions

Run /improve and codehalter reads back its session logs, spots where it stumbled or you corrected it, and proposes concrete edits to its own prompts and skills (PLAN.md, EXECUTE.md, …). You accept or skip each one, so it sharpens to your codebase over time.

// get started

Up and running in under 30 seconds.

codehalter is a single binary your editor launches over ACP. The clip is the full Zed setup, start to finish; the steps are below.

  1. Download the binary

    Run curl -sL https://raw.githubusercontent.com/tbocek/codehalter/main/install.sh | bash, or grab a prebuilt binary from the latest release (Linux / macOS; on Windows use WSL).

  2. Make sure Docker is running

    codehalter only ever runs inside a devcontainer, so it needs Docker (or Podman) up. Start Docker Desktop or dockerd before you launch it.

  3. Point it at your model

    Run codehalter --setup for the interactive LLM config, or edit ~/.config/codehalter/settings.toml and set server + model for any OpenAI-compatible endpoint.

  4. Add it to your editor

    In Zed's settings.json, add codehalter under agent_servers with "command": "codehalter", then pick it in the agent panel.

  5. Open a project, reopen in the container

    On first run it scaffolds a .devcontainer. Reopen the project inside it and start the plan, execute, verify, document loop.

Skill Crafter

Different models need different skills.

codehalter steers models with SKILL files — short lists of behavioral statements. But a statement only earns its place if the model would get it wrong without it: some behaviors are baked into a model's weights, others aren't, and a finetune shifts the line again. Skill Crafter measures which is which, per model, and prunes the rest. Statement proposals surface via /improve; broadly useful ones graduate into the shared skill base, and every statement then faces the probe.

01

Segment

A judge splits a skill into atomic claims — one testable behavior each, mapped to its exact source; a second judge repairs any citation the first got wrong.

02

Author

For each claim it writes a disguised question — usually "how would you approach this?" — plus a rubric for what a claim-following plan contains that an ignoring one lacks.

03

A / B probe

The target answers the question with the whole skill loaded and with nothing at all, several times over, to average out noise.

04

Judge & prune

If the model does it unprompted — drop the line. If the skill is what fixed it, keep it (keep-biased: drop only what needs no telling). Verdicts build a per-model skill file.

A single probe, concretely — claim: "in Go, wrap errors with fmt.Errorf and %w"

Answer A · no instruction
returns the error bare:
return err
— loses the call-site context.
Answer B · with the skill loaded
wraps with context:
return fmt.Errorf("load config: %w", err)
— chain preserved.
KEEP B satisfied the rubric, A did not — this model needs the line.

Saved input tokens are saved prefill time.

Skill files ride along in the prompt, so every statement is paid for in prefill: before the model emits its first token, it must process the entire input — that is what time-to-first-token mostly is, and it grows with prompt length (super-linearly on long prompts, because attention compares every token with every other). Prompt-compression research confirms the same claim from the other direction: LLMLingua reports 1.7–5.7× end-to-end speedups from shrinking prompts, with little quality loss.

This bites hardest exactly where codehalter lives: local models on modest hardware, where prefill throughput is a few hundred to a few thousand tokens per second. A few KB of statements a model never needed adds seconds before the first token — on every cache-miss turn, for the lifetime of the setup. A pruned skill is prefill you never pay again, and context head-room you get back.

  1. LLMLingua: Compressing Prompts for Accelerated Inference of LLMs — Jiang et al., EMNLP 2023: up to 20× compression, 1.7–5.7× speedup (Microsoft Research summary)
  2. Prompt Compression in the Wild: Measuring Latency, Rate Adherence, and Quality — direct latency measurements of compressed prompts
  3. NVIDIA NIM LLM benchmarking metrics — TTFT: the longer the prompt, the higher the TTFT; prefill is compute-bound
  4. IBM: Time to First Token — prompt length as a primary TTFT driver
  5. Redis: Prefill vs Decode — why input processing scales with (and beyond) prompt length

Help build the skill base.
