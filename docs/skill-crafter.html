codehalter · Skill Crafter

Different models need different skills.

codehalter steers a coding model with SKILL files: short lists of behavioral statements like "wrap errors with %w" or "never grep where a language server can answer". But a statement only earns its place if the model would get it wrong without it.

Some behaviors are baked into a model's weights; others aren't. A finetune shifts the line again. So the same skill file is redundant for one model and load-bearing for another. Skill Crafter measures which is which, per model, and prunes the rest.

The hypothesis

One prompt does not fit every model.

Gemma has different instincts than Qwen. A model already inclined to check every Go error gains nothing from being told to — that line is pure token cost, and on a smaller model a wall of redundant rules can even crowd out the instructions that matter. Meanwhile the exact same line might be the one thing keeping another model from swallowing errors.

My claim is simple: the right skill set is a property of the model, not the task. Instead of hand-tuning one prompt and hoping it generalises, we should test every statement against every model and keep only what changes behavior. The output is a set of lean, per-model skill files — each proven, statement by statement, to earn its keep.

How it works

Collect, probe, prune.

Skill proposals surface naturally: when you run /improve, codehalter mines the session for statements that would have helped. Broadly useful ones graduate into the shared skill base. From there, every statement faces the probe.

01

Segment

A judge model splits a skill file into atomic claims — one testable behavior each, mapped back to its exact source lines.

02

Author

For each claim it writes a disguised coding question plus a rubric: what a claim-following answer contains that a claim-ignoring one lacks.

03

A / B probe

The target model answers the question with and without the claim, several times over, to average out noise.

04

Judge & prune

If the two answers are alike, the model already knew — drop the line. If the claim fixed it, keep it. Verdicts build a per-model skill file.

A single probe, concretely — claim: "in Go, wrap errors with fmt.Errorf and %w"

Answer A · no instruction
return the error bare:
return err
— loses the call-site context.
Answer B · with the claim
wraps with context:
return fmt.Errorf("load config: %w", err)
— chain preserved.
KEEP B satisfied the rubric, A did not — this model needs the line.
Why it's good

Evidence, not vibes.

Prompt engineering is usually guesswork iterated by feel. Skill Crafter turns each line of the prompt into a falsifiable claim with a measured verdict.

Every line earns its place

A statement stays only if a model measurably behaves better with it. No cargo-culted rules, no "it probably helps".

Per-model skill sets

Gemma gets Gemma's file, Qwen gets Qwen's, each finetune its own. Leaner prompts, more room for the instructions that count.

It's just data

Every verdict is logged with both answers and the judge's reasoning. Disagree? Read the transcript, re-run, and see for yourself.

Call for participation

Help build the skill base.

This only works with breadth — many models, many finetunes, many probed statements. If you run local models and care about getting the most out of them, come measure with us.

  • Run the crafter against a model we don't have yet and open a PR with its skill set + stats.
  • Propose statements — hit /improve during real coding sessions and submit the fixes that would have helped.
  • Challenge a verdict — if a keep/drop looks wrong, the transcript is right there. Sharpen the probe.
  • Improve the method — better question authoring, sharper rubrics, less judge bias. It's all open.
